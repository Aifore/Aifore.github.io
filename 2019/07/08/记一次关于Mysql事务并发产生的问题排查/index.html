记一次关于Mysql事务并发产生的问题排查

发表于 2019-07-08 | 更新于 2020-07-19 | 分类于 MySQL | | 阅读次数：

线上用户的登录时会更新token，但会时不时报出token更新失败的异常，本遍主要记录该问题的排查处理过程。

一、服务端环境

  1. 表结构

    • user_id+token 联合主键（聚簇索引）；
    • user_id 唯一索引（辅助索引）；

    （不得不佩服前人的对这个表的索引设计）

  2. Django + sqlalchemy(Django自带的ORM不出产生这个问题)

  3. 该表只有一个方法对其进行写（不存在存在user_id的记录）或者更新（存在存在user_id的记录）

  4. orm.Model经过封装，每次调用该方法都会开启一个事务

二、客户端设计

  1. user_token_info接口的调用入口会防重复点击
  2. 弱网情况如果响应超时，会重试请求该接口

第二点因为网络原因，可能导致两个请求同时到达服务端，产生并发情况

那么就先想想这个事务是怎么加锁的？

user_id是唯一索引，user_id+token是联合主键，所以该表有两棵B+树（一个聚簇索引树a，一个辅助索引树b）。通过user_id执行查询时，会在树a上给user_id=5的索引加上X锁；因为user_id+token是联合主键，所以user_id是可能出现多条记录的（不考虑辅助的唯一索引），则会在user_id=5的两侧间隙加上间隙锁。

同样，在出现Mysql事务并发时，讨论两种情况：

  1. 该记录存在，能查找到

    执行查询sql后，树a上user_id=5的索引被加上X锁，(1, 5)(5, 9)两个间隙被加上了间隙锁（假如上一条记录是1，下一条是9）

    因为user_id被加上了X锁，所以事务B执行查询sql时，会等待事务A将锁释放才能继续执行。

    因此这种情况时不会产生异常的。

  2. 该记录不存在，会插入一条记录

    执行查询sql后，由于没有user_id=5的这条记录，所以没有X锁，只会在(1, 9)区间加上间隙锁

    由于间隙锁不是互斥的，所以两个并发事务没查询到记录时，会同时在(1, 9)区间加上间隙锁，然后其中任意一个事务执行插入sql插入一条user_id=5（或者此区间的任意一个数值）的记录时，会发现此区间被另一个事务的间隙锁锁住，从而导致循环等待对方释放锁，产生死锁。

既然是因为间隙锁导致的死锁，那么如何修复呢？这个在下面单独讨论。

四、优化

上面出现的第二个异常，产生死锁有两种可能，一种就是弱网导致客户端重试引起并发事务，另一种是插入间隙内的其他数值，所以也就解释了为什么第二个异常（两种情况都有）出现的频率还比第一个（只有第一种情况）稍微要高一些。

归根结底，其实只要是使用了for update当前读，不管使用什么索引，当记录查找不到时，都会使用间隙锁，所以单纯改索引的话无法解决第二种异常。

  1. 之所以会使用间隙锁，因为for update为当前读，防止该事务期间其他事务插入数据产生幻读。那么可以改成快照读，但由于这个表结构使用了联合主键，弱网时所以会产生第一个问题，所以还需要将联合主键改为user_id单一主键，或者使用自增主键，user_id设成唯一索引。

    • user_id单一主键：因为user_id是无序插入的的，所以每次插入都会为维护索引产生额外的开销，并且更容易产生页分裂，所以最好使用在读多写少的场景
    • 自增主键：插入时需要维护两棵B+树，但由于辅助索引的叶子结点是储存的id主键，所以页分裂的频次很低，查询时同样需要两次索引查询，速度稍慢（可以使用索引覆盖）

  2. 如果不考虑更改表的索引结构，那么只能使用快照读才能防止第二个异常出现（而且概率更大，宁愿报第一个异常，也不能报第二个），如何修复第一个异常，只能通过异常处理了。
