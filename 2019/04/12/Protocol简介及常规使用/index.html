Protocol Buffer Basics: Python

本教程提供了一个Python程序员使用protocol buffers的基本的入门教程。

通过创建一个简单的示例应用程序,它向您展示了如何

  • 在一个.proto文件中定义Message的格式。

  • 使用protocol buffer 编译器。

  • 使用Python protocol buffer API去读写Message。

这不是一个在Python中使用protocol buffers的一个全面的指南。有关更详细的参考信息，请阅读 Protocol Buffer Language Guide，Python API Reference，Python Generated Code Guide和Encoding Reference。

1. Why Use Protocol Buffers?

我们将使用的示例是一个非常简单的“地址簿”应用程序，可以从一个文件中读写人们的联系方式。地址簿中的每个人都有名字、ID、电子邮件地址和电话号码。

你如何序列化和检索这样的结构化数据？这里有一些办法可以解决这个问题：

  • 使用Python处理。这是默认方法，因为它内置于语言中，但它不能很好地处理模式演变，如果需要与使用C ++或Java编写的应用程序共享数据，也不能很好地工作。

  • 你可以发明一种特别的方式将数据项编码为一个字符串，如将4个int编码为“12:3:23:67”。这是一个简单的和灵活的方法，尽管它需要一次性编写编码和解析的代码，并为解析产生一定的运行成本。这最适合编码非常简单的数据。

  • 用XML序列化数据。这种方法非常有吸引力，因为XML具有易读性，还有了许多库，用来支持各种语言。如果你想和其它应用/工程共享数据，这是一个不错的选择。但是，XML也是出了名的耗空间，还有，编码/解码会令应用程序性能产生巨大的损失。加上操纵一个XML DOM树通常会比操纵类中的字段复杂。

Protocol buffers是灵活，高效，自动化的解决方案，可以解决这个问题。有了它，你就可以编写一个.proto文件用来描述你想存储的数据结构。因此，protocol buffer编译器会创建一个类，该类使用高效的二进制格式实现协议缓冲区数据的自动编码和解析。生成的类为构成Protocol buffers的字段提供getter和setter，并负责将Protocol buffers作为一个单元读取和写入的细节。重要的是， the protocol buffer format（协议缓冲区格式）支持随着时间的推移扩展格式的想法，使得代码仍然可以读取用旧格式编码的数据。

2. Defining Your Protocol Format

为了创建你的“地址簿”应用，你会用到一个.proto文件。.proto文件的定义很简单：你可以为你想序列化的每个数据结构添加一条Message，然后在Message中为每个字段指定名称和类型。

让我们来看看每个部分的作用：

  • 包的声明：这个.proto文件开头是包的声明，为了帮助防止在不同的工程中命名冲突。在Python中，包通常由目录结构决定的，所以这个由你的.proto文件定义的包，在你生成你代码中是没有效果的。但是，你应该坚持声明这条语句，为了在protocol Buffers的命名空间中防止名子的冲突，就像其它非Python的语言那样。
  • Message：一个Message是一个包含一组类型字段的集合。有许多简单的标准的数据类型可以用在类型字段中，包括bool，int32，float，double和string。
    • 你也可以使用更加多的结构来定义你的Message，例如用其它Message类型当作类型字段（在上面的例子PersonMessage中就包含了PhoneNumberMessage，还有AddressBookMessage包含PersonMessage）。
    • 你也可以定义Message嵌入其它的Message——PhoneNumber类型就是在Person类型中定义的。
    • 你也可以定义一个枚举类型，如果你想你其中一个字段有一个预设的类型列表——你可以将你的电话号码列举为MOBILE，HOME或者WORK。
  • 元素标签：那个“＝1”，“＝2”标记每个元素的识别，作为二进制编码中字段的唯一的标签。标签号1-15需要少于一个字节来编码而不是更高的数字，所以，作为最优化的方案，您可以决定将这些标签用于常用或重复的元素，把16或最高的数字留给不常用的optional元素。每个重复的字段里的元素要求重新编码它的标签数字，所以重复的字段特别适合使用这种优化。

每个字段一定要被以下的修饰语修饰：

  • required：必须提供该字段的值，否则这条Message会被认为“未初始化”。序列化未初始化的Message会出现异常。 解析一条没有初始化的Message会失败。除此之外，必填字段的行为与可选字段完全相同。
  • optional：可选字段可以设置也可以不设置 。如果没有设置值，会使用默认值。
    • 对于简单类型，您可以指定自己的默认值，就像我们type在示例中为电话号码所做的那样。如果不指定则使用系统默认值：数字类型为0，字符串为空字符串，bools为false。
    • 对于嵌入式消息，默认值始终是消息的“default instance”或“prototype”，其中没有设置其字段。如果调用访问器获取尚未显式设置的optional（或required）字段的值，将始终返回该字段的默认值。
  • repeated：这个字段会重复几次一些号码（包括0次）。重复值的顺序将保留在协议缓冲区中。重复的字段会被认为是动态大小的数组。

Required Is Forever 你应该非常小心地把字段标记为required。如果在某一时刻你希望停止写或发送一个必填字段，则将字段更改为可选字段会有问题——老的阅读器会认为没有这个字段Message是不完整的,而且可能会无意中拒绝或删除它们。你应该考虑为你的buffer编写特定于应用程序的自定义验证例程。一些Google工程师的结论是使用required弊大于利；他们更愿意只用optional和repeated。但是，这一观点并不普遍。

您.proto可以在Protocol Buffer Language Guide中找到编写文件的完整指南- 包括所有可能的字段类型。不要去寻找类继承类似的工具，但协议缓冲区不会这样做。

4.1 Enums：枚举

Enums是由metaclass扩展成的一组具有整数值的符号常量。例如常量addressbook_pb2.Person.WORK的值为2。

4.2 Standard Message Methods

每个Message类都包含一些其它方法，让你检查或操作整个message，包括：

  • IsInitialized(): 检查required字段是否都设置了值。

  • __str__(): 返回一个可读的message,对于调试尤其有用。(通常调用str(message)或print message。)

  • CopyFrom(other_msg): 用给出的message的值覆盖这个message

  • Clear(): 清除所有元素，回到空状态。

这些方法实现了Message接口。想了解更多，可以看Message完整的API文档。

4.3 Parsing and Serialization

最后，每个protocol buffer类都有通过二进制格式读/写message的方法，包括：

  • SerializeToString(): 序列化消息并以字符串的方式返回。 注意，字节是二进制的，而不是文本； 我们只使用str类型作为一个方便的容器。
  • ParseFromString(data): 从给出的字符串中解析一条message。

这些只是解析和序列化提供的几个选项。再次，请参阅Message API参考以获取完整列表。

Protocol Buffers和O-O Design Protocol buffer 类基本上是dumb data holders（类似于C++的structs）；它们在对象模型中没有成为first class citizens。如果你想为已生成的类添加丰富的行为，更好的方法是把已生成的protocol buffer类封装在一个特定于应用程序的类中。如果您无法控制.proto文件的设计（例如，如果您正在重用另一个项目中的一个），则封装Protocol Buffers也是一个好主意。在这种情况下，您可以使用封装类去制作更适合应用程序独特环境的接口：隐藏一些数据和方法，公开便利的函数等等。您永远不应该通过继承它们来向生成的类添加行为。这将打破内部机制，无论如何都不是良好的面向对象的实践。

7. Extending a Protocol Buffer

迟早你会发布使用protocol buffer的代码，毫无疑问你会想“改善” protocol buffer的定义。如果你想让你的新buffers 是向后兼容的，又或者你的旧buffers是向前兼容（能够前后兼容，至少不产生冲突导致以前的程序无法运行，谁都想这样设计吧😂），那么你需要遵守一些规则。在新版本的protocol buffer中：

  • 你不能更改任何现有字段的标签数字。
  • 你不能添加或删除任何required的字段。
  • 你可以删除optional 或repeated 字段。
  • 你可以添加新的optional 或repeated 字段，但必须使用新的标签数字。（即在protocol buffer从来没有使用过的标签数字，甚至不能使用已删除的标签数字）

这些规则有一些例外，但他们很少使用。

如果你遵守这些规则：

  • 旧的代码能够兼容地读取新messages，但会忽略所有新增字段；
  • 旧的代码被删除的可选optional 字段会使用它们的默认值，被删除的repeated字段会变为空；
  • 新代码能够兼容地读取旧messages；
  • 旧message不会显示新的optional字段，所以你需要明确地检查它们是否设置has_，或者检查你的.proto文件内，在标签数字后是否提供合理的默认值：[default = value]。如果没有为optional元素设置默认值，则使用特定于类型的默认值：对于字符串，默认值为空字符串；对于布尔类型，默认值为false；对于整型类型，缺省为0；
  • 如果你添加一个新repeated字段，你的新代码将无法判断它是否是空的(通过新代码)或从未设置(旧代码)，由于没有给它设置has_。（这句没看懂🤔）

8. 高级用法

Protocol buffers的用途不仅仅是简单的访问器和序列化。请务必浏览Python API参考，以了解您可以使用它们做些什么。

协议消息类提供的一个关键特性是映射。您可以迭代消息的字段并操纵它们的值，而无需针对任何特定的消息类型编写代码。使用映射的一种非常有用的方法是将协议消息转换为与其他编码（例如XML或JSON）之间的转换。更高级的映射使用可能是找到两个相同类型的消息之间的差异，或者开发一种“协议消息的正则表达式”，您可以在其中编写与某些消息内容匹配的表达式。如果您运用自己的想象力，可以将协议缓冲区应用于比您最初预期更广泛的问题！

映射只是Message接口提供的一部分功能。
