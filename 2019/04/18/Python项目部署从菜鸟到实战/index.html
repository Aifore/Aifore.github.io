Python项目部署从菜鸟到实战

发表于 2019-04-18 | 更新于 2019-07-14 | 分类于 运维 | | 阅读次数：

Python项目部署比较流行的架构为：

  • nginx：作为代理服务器提供web服务，主要作为是提供反向代理和负载均衡等功能；
  • supervisor：作为进程管理工具，统一管理服务器上的多个项目并提供项目状态监控的功能，并且在项目出现宕机时自动重启项目，保证服务的稳定性；
  • uWSGI/gunicron：作为uwsgi服务器，为python项目提供WSGI服务，并且通过多进程方式启动项目，提升项目服务性能；

1. 依赖安装

安装：

pip install supervisor ❣️ uwsgi ❣️ fabric==1.14.1

5. 注意事项

  • supervisor踩坑：
    • directory：directory=/Users/pander/Talent/shark/shark需要使用绝对路径，使用~/Talent/shark/shark的话会报错提示找不到路径；
    • command：command=<虚拟环境路径>/bin/uwsgi <conf.d的路径>/conf.d/<项目名>.uwsgi.ini匹配好命令与执行文件。我不小心把<虚拟环境路径>/bin/uwsgi中的二进制文件uwsgi写成了python2，导致无法执行<项目名>.uwsgi.ini而一直不能启动该django项目。
  • Nginx踩坑：
    • 起初项目内是配置的代理转发：proxy_pass http://127.0.0.1:3344;，请求接口始终报502错误，说明nginx与项目之前的连接不通，尝试将http://127.0.0.1:3344改成127.0.0.1:3344，重新加载nginx配置的时候就会报错：invalid URL prefix，看来代理转发proxy_pass只能是http://127.0.0.1:3344这种http请求格式。看了一下其他博客，Django的项目都是使用的uwsgi转发：uwsgi_pass 127.0.0.1:3344，并且包含了uwsgi参数：include uwsgi_params;，于是改成uwsgi转发并添加uwsgi_params，重新加载配置，请求接口成功。此时恍然大悟，Django使用uwsgi服务器是因为它使用的是uWSGI协议来处理请求，在处理请求之前必须将http请求转换成自己能够处理的uWSGI请求，所以使用uwsgi启动Django后，无法直接处理nginx转发的http请求，故而一直报502错误，nginx必须将http请求改为uWSGI请求再转发uwsgi服务器；
    • 两个server如果监听同一个端口，他们的server_name不能相同。
