记一次API json转protobuf重构

发表于 2020-02-22 | 更新于 2020-12-17 | 分类于 日常总结 | | 阅读次数：

背景

  1. 项目陈旧，至今未有大的优化重构

    申明：以下不带个人情绪，没有对初期设计者的任何鄙视😇。

    一般项目从早期创建到一两年的需求迭代后，在产品稳定前，抛开一个扩展性很好的优秀设计外，都很容易发展成以下这个样子。

    • 工程/代码逻辑组织结构混乱，结构和逻辑毫无层次；
    • 数据库schema定义随意，字断随处可见的共用、混用、乱用；
    • 大量陈旧垃圾代码夹杂在其中，很难准确判断哪些还在用哪些已弃用；
    • 代码质量极差，各种编写不规范，各种逻辑揉杂，业务逻辑混杂不清；
    • 大量已弃用接口，大量接口业务逻辑不清晰，难以使用，使用成本极高；
    • 接口协议混乱，部分字断早期定义过于随意，以及大量无用字断，影响正常业务逻辑；
    • 代码可维护性极差，出现问题极难排查，修复问题难度也很大，无法快速判断修改部分代码的影响范围；
    • 等等… 俗称程序员的代码屎山，每次打开工程，表情是这样的：🥶

  2. 上面提到的接口使用成本极高

    • 现有的接口中，大量字断已经弃用或者乱用，早已不符合最初设计的业务含义；
    • json定义的数据接口，在需求迭代过程中改动过于随意，不利于后续维护；
    • json接口缺乏可靠的文档，大量字断的设计初衷，无从查证；
    • 数据结构的定义或改动，与客户端沟通不便，而protobuf有更好的自解释性和更清晰的前后兼容变动；

  3. 性能/效率相对较差；

  4. 大量无用数据，宽带占用越来越高；

  5. 数据安全性很差，极易被破坏者利用或者攻击；

  6. 使用的python版本和django框架以及其他库的版本太老；

  7. 等等…

方案

  1. 所有与客户端交互接口，统一 json 转 protobuf；

  2. python版本由2.7升级至3.7；

  3. 所有数据进行加密签名(aes + sign)保证安全性；

  4. 工程结构/代码组织进行分层梳理改造；

  5. 由于数据统一使用protobuf作为载体，所以所有接口统一使用post请求；

  6. HTTP状态码统一返回200，具体业务异常码定义在Resp结构中，返回非200时则表示当前接口服务不可用；

  7. 因为不是重构为全新的平台，所以mysql/mongo/redis底层数据结构设计不作任何改动；

  8. 重构的服务单独部署，与老的服务共用底层数据，老版客户端使用老接口，新版客户端使用新接口，新老接口互不干涉，独立服务；

  9. 代码只作梳理重写，业务逻辑保持不变；

  10. 升级框架和依赖库版本；

  11. 为服务加入时区；

实施

这里主要对工程组织结构的改造进行描述说明，其他的重构想大多是具体功能的代码实现，不做累述。

老工程结构

以上简单示意性地展示了工程结构(只是伪结构)，有哪些问题？

  1. 目录结构没有业务划分，业务模块、通用模块揉杂在一起；
  2. 业务模块的定义边界模糊，每个模块的责任不够清晰明确；
  3. 业务代码与通用代码耦合在一起，难以区分，与低耦合背道而驰；
  4. 同一个业务的业务逻辑分散，工程的每个角落都能找到一点相关的代码，与高内聚背道而驰；
  5. 各模块的model和dto层逻辑到处分散，没有统一的管理方式；
  6. 部分模块划分，粒度太大，导致不同业务的逻辑揉杂在一起；
  7. 命名过于随意，不少词不达意或者过于笼统的名字；
  8. 很多不知所以的目录或者模块；
  9. 等等…

光是结构组织都已经很混乱了，代码组织就更不用说了，不举例描述了🥶

虽然问题一目了然，但是改造的思路还是比较明确，就是对模块就行重新组织，梳理业务逻辑，明确业务边界，抽出核心业务代码，一类业务一个模块，将原模块中交织耦合的代码剥离至对应模块统一管理，业务模块和通用模块严格隔离，重写逻辑代码，进行最小单元拆分，明确最小功能范围，避免每个单元的逻辑流程过长。

新工程结构

改造原则：

  1. 业务接口层API尽量不要写得太重，不要做太多业务逻辑处理，每个API尽量保证功能单一明确。
  2. 接口层主要处理请求参数检验，权限控制等，组合下层核心服务接口(RPC或者直接进程内方法调用)，并返回约定的结构数据。
  3. 中间的核心业务逻辑层，按业务模块划分，若后期抽离微服务，整个业务模块支持抽离成单独的微服务以提供RPC调用。
  4. 最底层是数据访问层，包括持久化数据存储或者高速缓存。每个核心业务服务只能操作相应的存储层数据，不能跨服务调用其它服务的数据层，若需要，则必须在每个业务模块的service中提供对外调用接口。
  5. 不能跨层调用，比如业务接口层直接调用数据访问层。

根据以上几个原则，主要是将服务进行分层，采用业务接口层 + 业务逻辑层 + 数据访问层三层架构的组织形式，严格控制每个模块的作用范围。遵守Django的project/app代码组织结构，每一个core业务模块，就类似于Django的一个app。

后记

采用三层架构的组织形式后，各个功能模块就一目了然了，代码逻辑也简洁明了，每一个函数、每一个模块、每一个目录的功能职责清晰明确。开发新需求能够更好地组织代码结构，在限定范围内进行，查找问题能够更快速精准地定位。当然，这不一定是最好的改造形式，在版本迭代过程中肯定还会陆续发现新的问题，需要我不断摸索，在实践中继续寻找更好的答案。共勉！
